## 1. 整体架构概述

### 1.1 设计原则

| 原则 | 说明 |
|------|------|
| **数据不重复** | 原始数据（专利/商标）全局只存一份，空间通过引用关系绑定数据，避免冗余 |
| **空间强隔离** | 每个空间通过 `space_id` 过滤，用户只能看到自己空间内的数据 |
| **检索统一入口** | 无论数据来自专利库、商标库还是用户上传，均通过统一检索层访问 |
| **多租户支持** | 企业空间支持 `org_id` 级别隔离，个人空间与企业空间权限模型独立 |
| **可扩展** | 数据源类型、AI 字段扩展均通过配置驱动，不修改核心表结构 |

### 1.2 架构分层

```
┌─────────────────────────────────────────────────┐
│                   应用层                          │
│     分析大屏 / 数据检索 / 智能分析 / 空间设置        │
└───────────────────┬─────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────┐
│              空间数据隔离层                        │
│   space_items（引用关系） + space_tags（标签）      │
│   每个空间独立视图，数据不拷贝                      │
└───────────────────┬─────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────┐
│              统一数据层（Data Lake）               │
│  ┌──────────────┐ ┌────────────┐ ┌────────────┐  │
│  │ PostgreSQL   │ │Elasticsearch│ │  Milvus   │  │
│  │ 结构化字段   │ │ 全文检索   │ │ 向量检索  │  │
│  └──────────────┘ └────────────┘ └────────────┘  │
└───────────────────┬─────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────┐
│                 外部数据源                         │
│     专利数据库    商标数据库    用户上传文件         │
└─────────────────────────────────────────────────┘
```

## 2. 数据分层设计

### 2.1 第一层：统一数据湖

原始数据入库后归属全局数据湖，任何空间均可引用，不因引用而复制。

| 存储 | 作用 | 典型字段 |
|------|------|---------|
| PostgreSQL | 结构化元数据主库 | 专利号、申请人、申请日、分类号、法律状态 |
| Elasticsearch | 全文检索 | 摘要、权利要求、说明书全文 |
| Milvus / pgvector | 语义向量检索 | 摘要/权利要求的 embedding 向量 |
| OSS / MinIO | 原始文件存储 | PDF 原文、图片附件 |

### 2.2 第二层：空间数据隔离

通过 `space_items` 关联表实现"虚拟分组"，核心设计：

- 同一条数据可被多个空间引用（不复制原始数据）
- 每个引用可独立附加标签、备注、分组
- 删除空间只删引用关系，不影响原始数据

### 2.3 第三层：AI 扩展字段

AI 模型对原始数据做字段扩展（技术分类、创新评分等），结果存入 `data_enrichments` 表，与原始数据关联但不覆盖原始字段。

## 3. 详细数据库表设计

### 3.1 用户与权限模块

#### `users` — 用户表

```sql
CREATE TABLE users (
    user_id         UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    username        VARCHAR(64)     NOT NULL UNIQUE,
    email           VARCHAR(255)    NOT NULL UNIQUE,
    password_hash   VARCHAR(255)    NOT NULL,
    display_name    VARCHAR(128),
    avatar_url      VARCHAR(512),
    status          SMALLINT        NOT NULL DEFAULT 1,  -- 1:正常 0:禁用
    created_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_users_email ON users(email);
```

#### `organizations` — 企业/组织表

```sql
CREATE TABLE organizations (
    org_id          UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    org_name        VARCHAR(128)    NOT NULL,
    org_code        VARCHAR(64)     UNIQUE,             -- 企业唯一识别码
    industry        VARCHAR(64),
    logo_url        VARCHAR(512),
    owner_user_id   UUID            NOT NULL REFERENCES users(user_id),
    status          SMALLINT        NOT NULL DEFAULT 1,
    created_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW()
);
```

#### `org_members` — 企业成员表

```sql
CREATE TABLE org_members (
    id              BIGSERIAL       PRIMARY KEY,
    org_id          UUID            NOT NULL REFERENCES organizations(org_id),
    user_id         UUID            NOT NULL REFERENCES users(user_id),
    role            VARCHAR(32)     NOT NULL DEFAULT 'viewer',
                                    -- admin / analyst / data_specialist / viewer
    joined_at       TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    invited_by      UUID            REFERENCES users(user_id),
    status          SMALLINT        NOT NULL DEFAULT 1,
    UNIQUE (org_id, user_id)
);

CREATE INDEX idx_org_members_org ON org_members(org_id);
CREATE INDEX idx_org_members_user ON org_members(user_id);
```

#### `role_permissions` — 角色权限配置表

```sql
CREATE TABLE role_permissions (
    id              BIGSERIAL       PRIMARY KEY,
    org_id          UUID            NOT NULL REFERENCES organizations(org_id),
    role            VARCHAR(32)     NOT NULL,
    resource        VARCHAR(64)     NOT NULL,  -- 'space' / 'dataset' / 'model' / 'member'
    action          VARCHAR(32)     NOT NULL,  -- 'read' / 'write' / 'delete' / 'share'
    is_allowed      BOOLEAN         NOT NULL DEFAULT TRUE,
    UNIQUE (org_id, role, resource, action)
);
```

## 4. 检索方案设计

### 4.1 三层检索架构

```
用户输入检索词（关键词 / 高级字段组合）
          │
          ▼
┌─────────────────────────────────────┐
│  第一层：空间范围过滤（PostgreSQL）   │
│  SELECT source_id FROM space_items  │
│  WHERE space_id = ? AND source_type = 'patent'  │
└────────────────┬────────────────────┘
                 │ 候选 ID 列表
                 ▼
┌─────────────────────────────────────┐
│  第二层：内容检索（Elasticsearch）   │
│  Filter by _id in 候选列表          │
│  Must: 关键词全文匹配                │
│  Should: 字段权重提升               │
└────────────────┬────────────────────┘
                 │ 命中文档 + 分数
                 ▼
┌─────────────────────────────────────┐
│  第三层：结果组装（PostgreSQL JOIN） │
│  关联 AI 扩展字段 / 空间标签 / 备注  │
└─────────────────────────────────────┘
```

### 4.2 检索字段权重（专利场景）

| 字段 | ES 字段名 | boost 权重 |
|------|----------|-----------|
| 标题 | title | 5.0 |
| 摘要 | abstract | 3.0 |
| 权利要求 | claims | 2.0 |
| 说明书 | description | 1.0 |
| 申请人 | applicant | 2.0 |
| IPC 分类号 | ipc_codes | 4.0 |

### 4.3 高级检索支持字段

```
申请人 AND (关键词1 OR 关键词2) NOT 关键词3
申请日期范围: 2020-01-01 ~ 2024-12-31
IPC 分类: H04L*
法律状态: 有效
国家/地区: CN / US / EP
```

### 4.4 语义检索（AI 模式）

用户输入自然语言 → 调用 embedding 模型生成向量 → 在 Milvus 检索最相似的 top-K 专利 → 用候选 ID 列表走空间过滤。

### 4.5 "从外部检索后添加到空间"的数据流

```sql
-- 步骤1: 检索外部专利数据库，结果入库（若不存在）
```

## 5. 数据流程说明

### 5.1 创建空间并绑定数据集

```
用户选择数据集 → 创建 space_datasets 记录
用户选择 AI 模型 → 创建 space_models 记录
系统生成 spaces 记录 → 返回 space_id
```

### 5.2 检索并收藏数据

```
1. 在数据集市检索（全局范围，不限空间）
2. 用户选中条目，点击「添加到空间」
3. 系统：
   a. 若该条目不在 patents / trademarks 表中 → 先入库
   b. 在 space_items 插入引用记录
   c. 用户可设置标签、备注
4. 数据立即在该空间的检索结果中可见
```

### 5.3 AI 字段扩展流程

```
1. 用户在空间中选择运行 AI 模型
2. 系统取出 space_items 中的所有 source_id
3. 批量调用 AI 模型（异步队列）
4. 结果写入 data_enrichments 表
5. 前端轮询或 WebSocket 推送进度
6. 扩展字段在数据字段页、分析大屏中可用
```

### 5.4 空间删除流程

```
1. 标记 spaces.status = 0（软删除）
2. 级联删除 space_items（CASCADE）
3. 不删除 patents / trademarks 原始数据
4. 保留 operation_logs 用于审计
```

## 6. 技术选型建议

### 6.1 分阶段选型

| 阶段 | 数据量 | 推荐方案 | 说明 |
|------|--------|---------|------|
| **MVP 阶段** | < 100 万条 | PostgreSQL 全量 + pg_trgm 扩展 | 无需额外中间件，全文检索够用 |
| **成长阶段** | 100万 ~ 1000万条 | + Elasticsearch 7.x/8.x | 独立检索服务，支持复杂查询 |
| **成熟阶段** | > 1000万条 | + Milvus 2.x 向量库 | 支持语义检索、AI 推荐 |

### 6.2 推荐技术栈

```
数据库:      PostgreSQL 15+（主库，含 pgvector 扩展）
检索:        Elasticsearch 8.x
向量:        Milvus 2.x 或 pgvector（初期）
缓存:        Redis 7.x（检索结果缓存、热点数据）
文件存储:    MinIO（私有化）或 阿里云 OSS
消息队列:    RabbitMQ / Kafka（AI 批处理异步任务）
后端框架:    Python FastAPI 或 Java Spring Boot
```

### 6.3 多租户隔离策略

| 隔离级别 | 方式 | 适用场景 |
|---------|------|---------|
| 行级隔离（推荐） | 所有查询强制带 `space_id` / `org_id` 过滤 | 当前阶段首选，实现简单 |
| Schema 隔离 | 每个企业独立 schema | 数据量大、合规要求高时考虑 |
| 数据库隔离 | 每个企业独立数据库 | 超高安全要求（金融/政务）时使用 |
